# LLM Wiki Skill

## 核心理念

传统 RAG 的问题：每次查询时重新检索和合成，知识不会积累。

LLM Wiki 的解决方案：LLM **增量构建和维护持久化 wiki** —— 一个结构化的、相互链接的 markdown 文件集合，位于你和原始源文件之间。

### 关键区别

| 传统 RAG | LLM Wiki |
|---------|----------|
| 查询时重新检索 | 一次性编译，持续维护 |
| 知识不积累 | 交叉引用已存在 |
| 每次重新推导 | 矛盾已被标记 |
| 临时答案 | 持续增长的合成 |

### 实践场景

- **个人**：追踪目标、健康、心理学、自我提升
- **研究**：深入某个主题数周或数月
- **读书**：逐章记录，构建角色、主题、情节线程的页面
- **团队**：内部 wiki，由 Slack 线程、会议记录、项目文档、客户通话驱动
- **其他**：竞品分析、尽职调查、旅行规划、课程笔记、爱好深度挖掘

## 三层架构

```
┌─────────────────────────────────────────────────────────────┐
│                        Raw Sources                           │
│  (不可变 - 你的源文件，LLM 只读不写)                          │
│  文章、论文、图片、数据文件                                    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                          The Wiki                            │
│  (LLM 拥有和维护 - 你读，LLM 写)                              │
│  摘要、实体页面、概念页面、对比、概览、综合                     │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                          The Schema                          │
│  (配置文件 - 告诉 LLM 如何结构化和维护 wiki)                   │
│  例如：CLAUDE.md、AGENTS.md                                  │
└─────────────────────────────────────────────────────────────┘
```

### Raw Sources（原始源文件）

- 你的精选源文档集合
- **不可变**：LLM 只读取，从不修改
- 这是你的事实来源
- 典型内容：文章、论文、图片、数据文件

### The Wiki（Wiki 层）

- LLM 生成的 markdown 文件目录
- **LLM 完全拥有这一层**：创建页面、更新页面、维护交叉引用、保持一致性
- 你读它；LLM 写它
- 典型内容：摘要、实体页面、概念页面、对比、概览、综合

### The Schema（配置层）

- 告诉 LLM wiki 如何结构化的文档
- 约定、工作流程
- 关键配置文件 —— 让 LLM 成为纪律性的 wiki 维护者，而非通用聊天机器人
- 你和 LLM 随时间共同演进

## 三大操作

### 1. Ingest（摄取）

将新源文件放入原始集合并告诉 LLM 处理它。

**典型流程**：
1. LLM 读取源文件
2. 与你讨论关键要点
3. 在 wiki 中写入摘要页面
4. 更新索引
5. 更新整个 wiki 中的相关实体和概念页面
6. 在日志中追加条目

单个源文件可能影响 10-15 个 wiki 页面。

**建议**：一次摄取一个源文件并保持参与 —— 你阅读摘要，检查更新，指导 LLM 强调什么。

### 2. Query（查询）

针对 wiki 提出问题。

**流程**：
1. LLM 搜索相关页面
2. 读取它们
3. 综合答案并引用

**答案形式**：
- Markdown 页面
- 对比表格
- 幻灯片（Marp）
- 图表（matplotlib）
- 画布

**关键洞察**：好的答案可以**作为新页面归档回 wiki**。你要求的对比、分析、发现的连接 —— 这些是有价值的，不应消失在聊天历史中。

### 3. Lint（健康检查）

定期要求 LLM 对 wiki 进行健康检查。

**检查项**：
- 页面之间的矛盾
- 过时声明（被新源文件取代）
- 孤立页面（没有入站链接）
- 提到但缺乏自己页面的重要概念
- 缺失的交叉引用
- 可以通过网络搜索填补的数据空白

Lint 让 wiki 在增长时保持健康。

## 索引和日志

### index.md（内容导向）

- wiki 中所有内容的目录
- 每个页面列出链接、一行摘要、可选元数据（日期、源计数）
- 按类别组织（实体、概念、源文件等）
- LLM 在每次摄取时更新它
- 回答查询时，LLM 先读取索引找到相关页面，然后深入研究

### log.md（时间导向）

- 发生了什么以及何时发生的仅追加记录
- 摄取、查询、lint 传递
- **提示**：如果每个条目以一致的前缀开始（例如 `## [2026-04-02] ingest | Article Title`），日志可以通过简单的 unix 工具解析：
  ```bash
  grep "^## \[" log.md | tail -5  # 给出最后 5 个条目
  ```
- 提供了 wiki 演化的时间线
- 帮助 LLM 理解最近做了什么

## 最佳实践

### 摄取建议

- **一次一个**：一次摄取一个源文件并保持参与
- **验证更新**：阅读摘要，检查更新，指导 LLM 强调重点
- **批量模式**：可以批量摄取多个源文件，但监督较少

### 查询建议

- **归档答案**：将有价值的答案作为新页面归档回 wiki
- **多种格式**：根据问题使用不同格式（markdown、表格、幻灯片、图表）

### 维护建议

- **定期 Lint**：定期进行健康检查
- **更新索引**：每次摄取后更新 index.md
- **记录日志**：在 log.md 中记录所有操作

## 工具集成

### Obsidian

- **Obsidian Web Clipper**：浏览器扩展，将 web 文章转换为 markdown
- **本地图片下载**：在设置中配置附件文件夹路径和快捷键
- **图形视图**：查看 wiki 的形状 —— 什么连接到什么
- **Marp 插件**：基于 markdown 的幻灯片格式
- **Dataview 插件**：查询页面 frontmatter

### 搜索

- **小规模**：index.md 文件足够
- **大规模**：考虑使用 qmd（本地 markdown 搜索引擎，混合 BM25/向量搜索）

## 为什么有效

维护知识库的繁琐部分不是阅读或思考 —— 是簿记。

- 更新交叉引用
- 保持摘要最新
- 注意新数据何时与旧声明矛盾
- 在几十个页面中保持一致性

人类放弃 wiki 是因为维护负担的增长速度超过价值。

**LLM 不会**：
- 感到厌烦
- 忘记更新交叉引用
- 可以在一次传递中接触 15 个文件

wiki 保持维护是因为维护成本接近零。

## 目录结构示例

```
your-wiki/
├── raw/                    # 原始源文件（不可变）
│   ├── sources/
│   │   ├── article1.pdf
│   │   ├── paper2.md
│   │   └── data3.csv
│   └── assets/             # 图片等附件
│
├── wiki/                   # LLM 维护的 wiki
│   ├── index.md           # 索引（内容目录）
│   ├── log.md             # 日志（时间记录）
│   ├── entities/          # 实体页面
│   ├── concepts/          # 概念页面
│   ├── summaries/         # 摘要页面
│   └── synthesis/         # 综合页面
│
└── CLAUDE.md              # Schema（配置文件）
```

## CLAUDE.md 示例配置

```markdown
# LLM Wiki 配置

## 目录结构

- `raw/` - 原始源文件（只读）
- `wiki/` - LLM 生成的 wiki（读写）
- `wiki/index.md` - 内容索引
- `wiki/log.md` - 操作日志

## 摄取流程

1. 读取源文件
2. 提取关键信息
3. 创建/更新摘要页面
4. 更新相关实体和概念页面
5. 更新 index.md
6. 在 log.md 中追加记录

## 查询流程

1. 读取 wiki/index.md
2. 找到相关页面
3. 综合答案并引用
4. 如果有价值，作为新页面归档

## Lint 流程

1. 检查页面之间的矛盾
2. 找出孤立页面
3. 识别缺失的交叉引用
4. 建议新的源文件或调查方向
```

## 开始使用

1. **创建目录结构**
2. **编写 CLAUDE.md 配置**
3. **摄取第一个源文件**
4. **查询和迭代**
5. **定期健康检查**
