# LLM Wiki Skill

> LLM 增量构建和维护持久化 wiki —— 一个结构化、相互链接的 markdown 文件集合。
> 核心区别：不同于 RAG 每次重新检索，wiki 是一次编译、持续维护的复合制品。

## 架构概览

```
Raw Sources（只读） → The Wiki（LLM 读写） → The Schema（配置）
  文章/论文/PDF         摘要/实体/概念/综合      CLAUDE.md
```

**关键路径**：
- 源文件：`personal-wiki/raw/sources/`（只读，LLM 不修改）
- Wiki 输出：`personal-wiki/wiki/{摘要卡,实体卡,概念卡,综合卡}/`
- 配置文件：`personal-wiki/CLAUDE.md`（定义摄取流程、页面格式、交叉引用规则）
- 索引：`personal-wiki/wiki/index.md`
- 日志：`personal-wiki/wiki/log.md`

## 意图识别

| 用户说 | 操作 | 执行 |
|--------|------|------|
| "摄取这个文档" / "整理到wiki" | Ingest | 执行摄取流程 |
| "查询wiki" / "wiki里有什么关于X的" | Query | 执行查询流程 |
| "健康检查" / "lint" / "检查wiki" | Lint | 执行健康检查流程 |
| "为X创建wiki页面" / "补充X的概念" | Ingest（部分） | 创建/更新指定页面 |

## 操作一：Ingest（摄取）

### 流程

```
源文件输入 → 提取 → 确认 → 构建 Wiki → 更新索引和日志
```

**Step 1：读取源文件**
- 输入：文件路径或 URL
- 文件路径：直接 Read 工具读取
- URL：依次尝试 `tavily_extract` → `web_reader` → `gstack browser`
- 全部失败 → 告知用户，提供备选方案（手动粘贴文本）

**Step 2：内容分析与确认 ⚠️ 必须执行**
- 展示提取到的核心要点（3-5 条）
- 展示识别到的实体列表和概念列表
- 让用户确认或调整重点方向
- 用户确认后才进入 Step 3

**Step 3：Wiki 网络构建**
- 创建摘要页面 → `wiki/摘要卡/{{SLUG}}.md`
- 创建/更新实体页面 → `wiki/实体卡/{{SLUG}}.md`
- 创建/更新概念页面 → `wiki/概念卡/{{SLUG}}.md`
- 更新已有页面的交叉引用
- **SLUG 命名规则**：中文标题直接用标题作为文件名（如`病毒循环.md`）；英文标题用小写kebab-case；去除 `: / \ | < > ? * "` 等文件系统非法字符

**Step 4：更新索引和日志**
- 更新 `wiki/index.md`：添加新页面条目
- 追加 `wiki/log.md`：记录操作详情（格式见下方）

### 摘要页面格式

```markdown
---
title: 页面标题
type: 摘要
date: YYYY-MM-DD
tags:
  - tag1
sources:
  - 源文件路径或标题
related:
  - 关联页面名
---

# 页面标题

## 摘要
一句话概括

## 核心要点
1. 要点一
2. 要点二

## 关联
- [[实体页面1]]
- [[概念页面1]]

## 来源
- 源文件路径
```

### 实体/概念页面格式

```markdown
---
title: 实体/概念名
type: 实体卡|概念卡
date: YYYY-MM-DD
tags:
  - tag1
related:
  - 关联页面名
---

# 实体/概念名

## 摘要
一句话概括

## 详细内容
...

## 关联
- [[关联页面1]]
- [[关联页面2]]
```

### 日志格式

```markdown
## [YYYY-MM-DD] ingest | 文章标题

**源文件**: `raw/sources/filename.md`
**操作**:
- 创建摘要: `wiki/摘要卡/filename.md`
- 更新实体: `wiki/实体卡/entity1.md`
**统计**:
- 新增页面: 1
- 更新页面: 2
```

## 操作二：Query（查询）

### 流程

```
用户提问 → 读索引 → 深入页面 → 综合答案 → 归档（可选）
```

**Step 1：读取索引**
- 读取 `wiki/index.md`，找到与问题相关的页面

**Step 2：深入页面**
- 读取相关页面全文
- 跟踪交叉引用（`[[wikilink]]`）发现更多关联内容

**Step 3：综合答案**
- 综合信息，引用具体来源页面
- 答案格式根据问题类型选择：
  - "是什么"/"定义"类问题 → 简洁回答 + 来源引用（1-3句）
  - "A和B有什么区别"类问题 → 表格对比
  - "怎么理解"/"深入分析"类问题 → 完整 markdown 页面（含摘要、要点、关联）
  - "帮我做个分享"类问题 → 幻灯片（Marp）

**Step 4：归档（可选）**
- 如果答案有长期价值，询问用户是否归档到 `wiki/综合卡/`

### 示例

```
用户："查询wiki：冷启动策略有哪些？"
→ 读 index.md 找到相关概念页面
→ 读 概念卡/冷启动策略.md、案例卡/xxx.md
→ 综合答案，引用 [[冷启动策略]] 页面
→ 询问：是否归档到 wiki/综合卡/？
```

## 操作三：Lint（健康检查）

### 流程

```
扫描 wiki → 检查各项 → 生成报告 → 记录日志
```

**Step 1：全量扫描**
- 扫描 `wiki/` 下所有 `.md` 文件
- 读取 `wiki/index.md` 获取完整页面清单

**Step 2：逐项检查**

| 检查项 | 方法 | 修复建议 |
|--------|------|---------|
| 矛盾声明 | 用 Grep 搜索 wiki/ 下相同实体名/概念名，对比相关页面的摘要段；如发现"A页面说X是Y"与"B页面说X不是Y"，标记为矛盾 | 标记矛盾，建议用户核实后更新 |
| 孤立页面 | 检查哪些页面没有被其他页面 `[[引用]]`：Grep 搜索每个页面名，统计入站引用数为0的页面 | 建议添加入站链接 |
| 缺失引用 | Grep 搜索正文中 `[[xxx]]` 格式的引用，检查 xxx.md 是否存在于 wiki/ 目录下 | 建议创建缺失页面 |
| 过时内容 | 对比 `log.md` 中最近 ingest 的摘要卡与已有概念卡，检查是否有新内容覆盖旧描述 | 建议归档或更新 |
| Frontmatter 完整性 | 检查每个页面的 `---` 之间的 YAML 是否包含 title、type、date、tags、sources、related 字段 | 补全缺失字段 |

**Step 3：生成报告**
- 输出结构化报告：通过/警告/错误三级
- 每个问题给出具体文件路径和修复建议

**Step 4：记录日志**
- 在 `wiki/log.md` 追加 lint 记录

## 边界条件

- **URL 无法访问** → 依次尝试 tavily_extract → web_reader → gstack，全部失败则报错
- **源文件不存在** → 告知用户，检查路径
- **提取不到有价值内容** → 告知用户，建议直接存为信息卡
- **Wiki 页面已存在** → 更新而非覆盖（追加新关联，保留已有内容）
- **大量页面(>10个)** → 分批展示确认，避免信息过载
- **索引文件不存在** → 先创建 `wiki/index.md` 骨架
- **日志文件不存在** → 先创建 `wiki/log.md` 骨架

## Frontmatter 规范

1. **不要在 frontmatter 内使用 `[[wikilink]]` 语法** — 用纯字符串
2. **使用 YAML 列表格式**（`- item`）而非内联数组（`[a, b]`）
3. **不要在 frontmatter 内放 Markdown 标题或注释**

## 最佳实践

- **一次摄取一个源文件**，保持参与，验证质量
- **定期 Lint**，保持 wiki 健康
- **归档有价值的查询答案**，避免知识流失
- **所有操作记录在 log.md**
