# Lobster Article - 文章处理

抓取网页文章内容，提取正文，以markdown格式存入 raw/sources ，并使用 AI 自动拆分为知识卡片。

## When to Use

当用户说：
- "帮我处理这个 URL"
- "拆解这篇文章: https://xxx"
- "提取这篇文章的判断卡"
- "把这个链接的内容存到知识库"
- "总结这篇文章的精华"
- "提取核心观点"

## How It Works

1. **URL 内容抓取**
   - 优先使用 `tavily_extract` MCP 工具获取网页内容
   - 如失败，尝试 `web_reader`
   - 如仍失败，尝试 `gstack` browser skill
   - **全部失败** → 告知用户并提供备选方案（手动粘贴文本、换其他工具）
   - 提取正文，过滤广告和导航
   - **备份源文件**（URL 摄取成功后必须执行）：
     - 将提取到的 markdown 正文保存到 `personal-wiki/raw/sources/{文章标题}.md`
     - 如果页面包含图片、视频、音频、PDF、Word 等附件资源，下载并保存到 `personal-wiki/raw/assets/{类型}/`（类型为 `images`/`videos`/`audio`/`pdf`/`documents`）
     - 在 `raw/sources/{文章标题}.md` 头部记录原始 URL 和备份时间

2. **AI 拆卡**（严格遵循下方拆卡方法论）
   - 分析文章内容
   - 按判别树分类为判断、方法、案例、信息
   - 按颗粒度标准控制粗细
   - 为每种类型创建对应卡片
   - **如果用户指定了重点类型**（如"重点提取判断卡"），优先提取该类型，其他类型可选

3. **展示预览并确认** ⚠️ 必须执行
   - 展示拟创建的卡片清单：类型、标题、一句话摘要
   - 让用户确认：哪些卡片创建、哪些跳过、是否需要调整
   - 用户确认后才执行保存

4. **保存到知识库**
   - 自动生成 frontmatter（遵循 `personal-wiki/CLAUDE.md` 格式规范）
   - 保存到 `personal-wiki/卡片库/{类型}/` 目录
   - 添加原文链接到 `sources` 字段

5. **保存后校验** ⚠️ 每次保存必须执行
   - 逐张卡片检查以下项目，不通过立即修复：

   **frontmatter 字段检查**：
   - `sources`: 只写来源标题（对应 `raw/sources/` 文件名），**不带 URL**
     - ✅ `普通人能不能成为张雪`
     - ❌ `普通人能不能成为张雪？ https://mp.weixin.qq.com/s/...`
   - `related`: 同批拆出的卡片必须交叉关联，**不允许为空列表**
     - 同一文章拆出 3 张卡，每张卡的 `related` 至少包含其他卡片之一
     - 检查语义关联性，不要强行关联无关卡片
   - `wiki_concepts`: 有 Wiki 模式时填入，纯拆卡模式可为空列表

   **正文章节检查**：
   - 必须包含 `## 相关卡片` 章节，用 `[[wikilink]]` 链接到 related 中的卡片
   - 必须包含 `## 来源` 章节，用 `[[wikilink]]` 链接到 `raw/sources/` 中的源文件
   - 不允许出现空的章节标题（如 `## 相关 Wiki 概念` 后无内容则删除该节）

   **交叉一致性检查**：
   - frontmatter 中的 `related` 列表 ↔ 正文 `## 相关卡片` 的 `[[wikilink]]` 列表必须一一对应
   - frontmatter 中的 `sources` ↔ 正文 `## 来源` 的 `[[wikilink]]` 必须对应
   - 不允许出现"frontmatter 有但正文没有"或"正文有但 frontmatter 没有"的不一致

## 边界条件

- **URL 无法访问** → 依次尝试 tavily_extract → web_reader → gstack，全部失败则报错
- **URL 指向非文章**（视频页、登录墙、404）→ 告知用户并建议手动提供文本
- **提取不到有价值卡片** → 告知用户文章内容不适合拆卡，建议直接存为信息
- **同名卡片已存在** → 询问用户：覆盖 / 新建（加后缀）/ 跳过
- **用户未提供 URL** → 询问文章来源（URL 或本地文件路径）

## 拆卡方法论 - 三大黄金法则

> **核心原则**：本 skill 严格执行三大黄金法则，所有拆卡决策必须通过这三条检验。
> 权威来源：`personal-wiki/wiki/综合卡/拆卡方法论.md`

### 📏 法则一：精确检索原则（拒绝长篇大论）

**检验标准**：
- **标题测试**：标题必须是**完整陈述句**，能在搜索结果中独立识别核心主张
- **关键词密度**：每张卡片只包含**一个核心概念词**，避免语义分散
- **检索意图对齐**：标题必须匹配用户可能的搜索查询
- **长度限制**：卡片正文不超过 300 字，超过则拆分

**反模式**：
- ❌ "关于产品增长的几点思考" → 太泛，无法精确检索
- ❌ "SBTI案例分析与传播策略总结" → 混合多个概念
- ✅ "传播力大于功能完备性" → 精确匹配检索意图
- ✅ "三步传播产品设计心法" → 明确方法，可检索

---

### 🎯 法则二：语义纯粹原则（保持卡片纯粹）

**检验标准**：
- **单一语义类型**：一张卡只回答一个问题（判断/方法/案例/信息）
- **避免混合**：案例中的方法必须拆分，数据中的解读必须分离
- **概念边界清晰**：不使用"以及"、"另外"、"同时"等连接词
- **可独立理解**：离开原文上下文，卡片仍能自足

**反模式**：
- ❌ 案例卡中内嵌方法步骤 → 拆成两张卡
- ❌ 判断卡中混杂数据引用 → 数据归案例，判断保留主张
- ❌ "传播很重要，而且要降低门槛" → 拆成两张独立卡

**执行规则**：
```
遇到混合内容 → 立即拆分
1. 识别主类型（判断/方法/案例）
2. 提取嵌套的其他类型
3. 用 related 字段建立链接
```

---

### 🔮 法则三：未来复用原则（剥离特定语境）

**检验标准**：
- **去语境化**：删除"这篇文章"、"作者认为"、"在XX领域"等限定词
- **抽象化表达**：将具体场景抽象为通用原则
- **适用条件前置**：把特定场景转化为"适用条件"字段
- **来源元数据化**：将来源信息放入 frontmatter 的 sources 字段

**反模式**：
- ❌ "根据本文作者的观点，短视频在2026年仍有机会"
- ✅ "垂直小号在2026年仍有流量红利"
   - 适用条件：C端内容产品、冷启动阶段
   - 来源：[具体文章链接放metadata]

**抽象化技巧**：
```
具体 → 通用
"抖音的算法推荐机制" → "算法推荐产品的冷启动策略"
"Nike的营销案例" → "品牌危机下的营销重启方法论"
```

### 三大法则的综合检验流程

```
每创建一张卡，通过三层检验：

Layer 1: 精确检索检验
├─ 标题是完整陈述句吗？
├─ 包含几个核心概念？(>1则拆分)
└─ 长度是否 ≤ 300字？

Layer 2: 语义纯粹检验
├─ 只回答一个问题吗？
├─ 是否混合多种类型？
└─ 离开原文能独立理解吗？

Layer 3: 未来复用检验
├─ 是否包含特定语境词？
├─ 能否抽象为通用原则？
└─ 来源信息是否在metadata？

✓ 三层全通过 → 创建卡片
✗ 任一层失败 → 重构或拆分
```

### 一、卡片分类规则（基于三大法则的细化）

#### 1.1 核心判别：每张卡回答一个问题

| 卡类型 | 回答的问题 | 核心特征 | 一句话验证 |
|--------|-----------|---------|-----------|
| **判断卡** | "该怎么**判断**？" | 可辩护的**主张**，有适用条件和边界 | "有人会不同意吗？" → 是 = 判断卡 |
| **方法卡** | "具体**怎么做**？" | 可复用的**流程**，有步骤和预期产出 | "别人按这个能做出来吗？" → 是 = 方法卡 |
| **案例卡** | "**发生了**什么？" | 有**叙事结构**的实例，有因果和教训 | "能追溯到具体来源吗？" → 是 = 案例卡 |
| **信息卡** | "**是什么**？" | 可检索的**事实**，中性无主张 | "搜30秒就能找到吗？" → 是 = **别建卡** |

#### 1.2 各类型必含字段

**判断卡**：主张 + 适用条件 + 不适用条件 + 信心等级

**方法卡**：前置假设 + 分步流程 + 预期产出 + 已知局限

**案例卡**：具体情境（谁/何时/何地）+ 发生了什么 + 为什么重要 + 来源可追溯

**信息卡**：极少创建，仅当"难搜索 + 高频引用 + 战略重要"三者同时满足

#### 1.3 提取优先级

```
判断卡 > 方法卡 > 案例卡 >>> 信息卡（信息卡默认不建）
```

#### 1.4 边界决策树

```
这段内容的核心目的？
│
├─ 帮我"做决定"（有主张、可辩论）
│  └─ 有适用条件吗？ → 判断卡
│  └─ 没有条件边界 → 不是有效判断，不建卡
│
├─ 告诉我"怎么做"（有步骤、可重复）
│  └─ 能跨场景复用吗？ → 方法卡
│  └─ 只针对某个具体场景 → 案例卡
│
├─ 展示"发生了什么"（有叙事、有因果）
│  └─ 有可提取的教训？ → 案例卡
│  └─ 只是个事实 → 信息卡（或丢弃）
│
└─ 只是陈述"是什么"（中性事实）
   └─ 搜索不到 + 高频引用 → 信息卡
   └─ 否则 → 不建卡
```

#### 1.5 常见边界 Case 速查表

| 模糊内容 | 关键判断 | 归类 |
|---------|---------|------|
| "短视频比长视频更适合冷启动" | 可辩论的主张 | **判断卡** |
| "用 5 步法做冷启动测试" | 可复用流程 | **方法卡** |
| "SBTI 从 0 到 10 万用户的 3 个阶段" | 特定实例 + 因果 | **案例卡** |
| "TikTok 日活 15 亿" | 搜 30 秒就有 | **不建卡** |
| 案例中内嵌的方法 | 可独立复用 | **拆！案例归案例，方法单独抽** |
| 既像判断又像方法 | 主要帮你"决定"还是"执行"？ | **按主用途归类** |
| 既像案例又像信息 | 有叙事/因果/教训吗？ | **有 = 案例，没有 = 丢弃** |
| 数据 + 解读 | 数据是证据，解读是判断 | **拆！数据归案例，解读归判断** |
| 名人名言 | 有独立洞察价值吗？ | **有 = 融入判断卡；无 = 丢弃** |

#### 1.6 多类型归属的 3 条处理原则

**原则 1：先拆再分** — 拆成两张卡各归各类，不用复合类型

**原则 2：按最高价值归类** — 判断卡 > 方法卡 > 案例卡 > 信息卡

**原则 3：交叉链接** — 用 `related` 和 `wiki_concepts` 建立关联

### 二、颗粒度标准（基于三大法则的细化）

#### 2.1 "刚刚好"的 5 条规则（更新版）

**规则 1：标题测试**（强化精确检索）
- 标题必须是**完整陈述句**，能在搜索结果中独立识别
- 标题长度：10-25字最佳，不超过30字
- 必须包含**核心概念词**，避免"关于"、"几点"等泛化词

**规则 2：一屏原则**（精确检索保障）
- 卡片正文不超过 **300 字**（比之前更严格）
- 超过300字 → 必须拆分或精简
- 每段不超过3句话，保持信息密度

**规则 3：独立性原则**（语义纯粹保障）
- 卡必须**自足**——不依赖其他卡就能被理解
- 不使用"上文提到的"、"正如前述"等指代词
- 所有引用内容用 `[[wikilink]]` 明确链接

**规则 4：社交发布测试**（未来复用保障）
- 这张卡发到社交媒体上，陌生人能看懂吗？
- 看不懂 → 太依赖上下文，需要重写
- 这张卡一年后你还能理解吗？不能 → 需要更多语境抽象

**规则 5：一个主张**（语义纯粹保障）
- 并列关系（第一、第二、第三）如果每点能独立成立，就拆
- 使用"以及"、"另外"、"同时"的地方是拆分信号
- 每张卡只回答一个核心问题

#### 2.2 颗粒度的两端信号

**太粗了 → 该拆**：
- 标题不能一句话概括
- 包含多个不相关的观点
- 需要"第一、第二、第三"来组织
- 超过一屏需要滚动
- 内容同时回答多个问题

**太碎了 → 该合并**：
- 离开上下文就失去意义
- 短于一个完整的句子
- 拆出来的碎片之间没有链接，变成孤儿卡
- 维护成本 > 价值

#### 2.3 实操原则

> "宁可先粗后细，不要一步到位拆太碎"

- 拆粗了 → 以后随时可以再拆，成本低
- 拆碎了 → 孤儿卡片维护成本高，且容易丢失

#### 2.4 Andy Matuschak 三步拆法

1. **写大卡**：抓住一簇内容的"大思想"。如果有多个大思想，就写多张卡（保持原子性）
2. **写细卡**：逐段扫描，提炼更精细的原子洞察
3. **建立连接**：搜索已有卡片，建立链接、合并、修订

### 三、反模式清单（基于三大法则的细化）

| 反模式 | 违反法则 | 为什么错 | 正确做法 |
|--------|---------|---------|---------|
| 原文摘抄不改写 | 法则3 | 没经过大脑，等于没读；包含过多特定语境 | 用自己的话重写，抽象为通用原则 |
| 标题写"关于XX" | 法则1 | 无法精确检索，语义模糊 | 标题用完整陈述句概括核心观点 |
| "以及"、"另外"连接 | 法则2 | 混合多个语义，不纯粹 | 立即拆分成独立卡片 |
| 案例中内嵌方法 | 法则2 | 语义不纯粹，无法独立复用 | 拆！案例归案例，方法单独抽 |
| "本文作者认为" | 法则3 | 特定语境，阻碍未来复用 | 去除限定词，来源放metadata |
| 超过300字的长卡 | 法则1 | 信息密度低，检索困难 | 拆分或精简，保持≤300字 |
| 每段都建信息卡 | 法则2 | 信息是商品化的，无沉淀价值 | 信息卡默认不建 |
| 不建链接的孤立卡 | 法则2 | 孤儿卡是Zettelkasten最危险的产物 | 每张卡至少链接2个其他卡或Wiki页面 |
| "第一、第二、第三" | 法则2 | 并列关系说明该拆，语义不纯粹 | 拆成多张独立卡 |
| 按来源而非概念组织 | 法则3 | 来源是上下文，概念才是知识 | 按概念原子组织，来源放metadata |

**未来复用检查**：
- [ ] 没有"本文作者"、"这篇文章"等限定词

### 四、Frontmatter 格式规范

创建卡片时必须遵循 `personal-wiki/CLAUDE.md` 的格式规范：

1. **不要在 frontmatter 内使用 `[[wikilink]]` 语法** — 所有属性值必须使用纯字符串
2. **不要在 frontmatter 内放 Markdown 标题** — frontmatter `---` 之间必须是合法 YAML
3. **使用 YAML 列表格式**（`- item`）而非内联数组（`[a, b]`）
4. **文件命名**：`{{SLUG}}.md`，日期和类型放 frontmatter 不放文件名

## Output

- 判断卡数量和内容
- 方法卡数量和内容
- 案例卡数量和内容
- 信息卡数量和内容（通常为 0）
