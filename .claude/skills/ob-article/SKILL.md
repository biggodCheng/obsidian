# Lobster Article - 文章处理

抓取网页文章内容，提取正文，并使用 AI 自动拆分为知识卡片。

## When to Use

当用户说：
- "帮我处理这个 URL"
- "拆解这篇文章: https://xxx"
- "提取这篇文章的判断卡"
- "把这个链接的内容存到知识库"

## How It Works

1. **URL 内容抓取**
   - 使用 web reader 或 browser 获取网页内容
   - 提取正文，过滤广告和导航

2. **AI 拆卡**（严格遵循下方拆卡方法论）
   - 分析文章内容
   - 按判别树分类为判断、方法、案例、信息
   - 按颗粒度标准控制粗细
   - 为每种类型创建对应卡片

3. **保存到知识库**
   - 自动生成 frontmatter（遵循 CLAUDE.md 格式规范）
   - 保存到 notes 目录
   - 添加原文链接

## 拆卡方法论

> 权威来源：`personal-wiki/wiki/synthesis/拆卡方法论.md`
> 本 skill 是该方法论的执行层，所有规则以 synthesis 文档为准。

### 一、卡片分类规则

#### 1.1 核心判别：每张卡回答一个问题

| 卡类型 | 回答的问题 | 核心特征 | 一句话验证 |
|--------|-----------|---------|-----------|
| **判断卡** | "该怎么**判断**？" | 可辩护的**主张**，有适用条件和边界 | "有人会不同意吗？" → 是 = 判断卡 |
| **方法卡** | "具体**怎么做**？" | 可复用的**流程**，有步骤和预期产出 | "别人按这个能做出来吗？" → 是 = 方法卡 |
| **案例卡** | "**发生了**什么？" | 有**叙事结构**的实例，有因果和教训 | "能追溯到具体来源吗？" → 是 = 案例卡 |
| **信息卡** | "**是什么**？" | 可检索的**事实**，中性无主张 | "搜30秒就能找到吗？" → 是 = **别建卡** |

#### 1.2 各类型必含字段

**判断卡**：主张 + 适用条件 + 不适用条件 + 信心等级

**方法卡**：前置假设 + 分步流程 + 预期产出 + 已知局限

**案例卡**：具体情境（谁/何时/何地）+ 发生了什么 + 为什么重要 + 来源可追溯

**信息卡**：极少创建，仅当"难搜索 + 高频引用 + 战略重要"三者同时满足

#### 1.3 提取优先级

```
判断卡 > 方法卡 > 案例卡 >>> 信息卡（信息卡默认不建）
```

#### 1.4 边界决策树

```
这段内容的核心目的？
│
├─ 帮我"做决定"（有主张、可辩论）
│  └─ 有适用条件吗？ → 判断卡
│  └─ 没有条件边界 → 不是有效判断，不建卡
│
├─ 告诉我"怎么做"（有步骤、可重复）
│  └─ 能跨场景复用吗？ → 方法卡
│  └─ 只针对某个具体场景 → 案例卡
│
├─ 展示"发生了什么"（有叙事、有因果）
│  └─ 有可提取的教训？ → 案例卡
│  └─ 只是个事实 → 信息卡（或丢弃）
│
└─ 只是陈述"是什么"（中性事实）
   └─ 搜索不到 + 高频引用 → 信息卡
   └─ 否则 → 不建卡
```

#### 1.5 常见边界 Case 速查表

| 模糊内容 | 关键判断 | 归类 |
|---------|---------|------|
| "短视频比长视频更适合冷启动" | 可辩论的主张 | **判断卡** |
| "用 5 步法做冷启动测试" | 可复用流程 | **方法卡** |
| "SBTI 从 0 到 10 万用户的 3 个阶段" | 特定实例 + 因果 | **案例卡** |
| "TikTok 日活 15 亿" | 搜 30 秒就有 | **不建卡** |
| 案例中内嵌的方法 | 可独立复用 | **拆！案例归案例，方法单独抽** |
| 既像判断又像方法 | 主要帮你"决定"还是"执行"？ | **按主用途归类** |
| 既像案例又像信息 | 有叙事/因果/教训吗？ | **有 = 案例，没有 = 丢弃** |
| 数据 + 解读 | 数据是证据，解读是判断 | **拆！数据归案例，解读归判断** |
| 名人名言 | 有独立洞察价值吗？ | **有 = 融入判断卡；无 = 丢弃** |

#### 1.6 多类型归属的 3 条处理原则

**原则 1：先拆再分** — 拆成两张卡各归各类，不用复合类型

**原则 2：按最高价值归类** — 判断卡 > 方法卡 > 案例卡 > 信息卡

**原则 3：交叉链接** — 用 `related` 和 `wiki_concepts` 建立关联

### 二、颗粒度标准

#### 2.1 "刚刚好"的 5 条规则

**规则 1：标题测试** — 标题必须是完整陈述句，能概括整张卡核心观点。如果标题只能写"关于 XX 的几点思考"，说明卡太粗了。

**规则 2：一屏原则** — 卡的内容不应超过一个屏幕（约 200-500 字中文）。需要滚动的卡 = 该拆。

**规则 3：独立性原则** — 卡必须自足——不依赖其他卡就能被理解。

**规则 4：社交发布测试（Thread 测试）** — 这张卡原封不动发到社交媒体上，别人看得懂吗？看不懂 = 重写或太碎。

**规则 5：一个主张** — 并列关系（第一点、第二点、第三点）如果每点都能独立成立，就该拆成多张卡。

#### 2.2 颗粒度的两端信号

**太粗了 → 该拆**：
- 标题不能一句话概括
- 包含多个不相关的观点
- 需要"第一、第二、第三"来组织
- 超过一屏需要滚动
- 内容同时回答多个问题

**太碎了 → 该合并**：
- 离开上下文就失去意义
- 短于一个完整的句子
- 拆出来的碎片之间没有链接，变成孤儿卡
- 维护成本 > 价值

#### 2.3 实操原则

> "宁可先粗后细，不要一步到位拆太碎"

- 拆粗了 → 以后随时可以再拆，成本低
- 拆碎了 → 孤儿卡片维护成本高，且容易丢失

#### 2.4 Andy Matuschak 三步拆法

1. **写大卡**：抓住一簇内容的"大思想"。如果有多个大思想，就写多张卡（保持原子性）
2. **写细卡**：逐段扫描，提炼更精细的原子洞察
3. **建立连接**：搜索已有卡片，建立链接、合并、修订

### 三、反模式清单（不要做）

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| 原文摘抄不改写 | "摘抄病"——没经过大脑，等于没读 | 用自己的话重写，费曼学习法 |
| 原子性 zealot | 拆太碎导致孤儿卡泛滥，维护成本 > 价值 | 宁可粗一点，保证独立性 |
| 每段都建信息卡 | 信息是商品化的，无沉淀价值 | 信息卡默认不建 |
| 不建链接的孤立卡 | 孤儿卡是 Zettelkasten 最危险的产物 | 每张卡至少链接 2 个其他卡或 Wiki 页面 |
| 标题写"关于 XX" | 标题 = API，必须可检索可理解 | 标题用完整陈述句概括核心观点 |
| 按来源而非概念组织 | 来源是上下文，概念才是知识 | 按概念原子组织，来源放 metadata |

---

### 四、Frontmatter 格式规范

创建卡片时必须遵循 `personal-wiki/CLAUDE.md` 的格式规范：

1. **不要在 frontmatter 内使用 `[[wikilink]]` 语法** — 所有属性值必须使用纯字符串
2. **不要在 frontmatter 内放 Markdown 标题** — frontmatter `---` 之间必须是合法 YAML
3. **使用 YAML 列表格式**（`- item`）而非内联数组（`[a, b]`）
4. **文件命名**：`{{SLUG}}.md`，日期和类型放 frontmatter 不放文件名

需要以下依赖之一：
- `web_reader` MCP server（推荐）

- 或 `gstack` browser skill

## Usage Examples

```
帮我处理这篇文章: https://example.com/article
拆解这个 URL 的内容
提取 https://xxx 的核心判断
```

## Output

返回创建的卡片列表：
- 判断卡数量和内容
- 方法卡数量和内容
- 案例卡数量和内容
- 信息卡数量和内容（通常为 0）
