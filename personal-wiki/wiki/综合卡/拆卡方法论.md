## 三大黄金法则（2026-04-13 更新）

> **核心理念**：拆卡的本质是构建可复用的知识资产。三大黄金法则确保每张卡片都能在未来被精确检索、独立理解、跨场景复用。

### 📏 法则一：精确检索原则（拒绝长篇大论）

**定义**：每张卡片必须能被独立检索到，标题是检索的API。

**检验标准**：
- **标题长度**：10-25字最佳，不超过30字
- **标题格式**：必须是完整陈述句，概括核心观点
- **关键词密度**：每张卡只包含一个核心概念词
- **内容长度**：正文不超过300字
- **禁用词**：禁止使用"关于"、"几点"、"思考"等泛化词

**反模式示例**：
- ❌ "关于产品增长的几点思考" — 太泛，无法精确检索
- ❌ "SBTI案例分析与传播策略" — 混合多个概念
- ✅ "传播力大于功能完备性" — 精确，可检索
- ✅ "三步传播产品设计心法" — 明确方法，可复用

**Why（为什么）**：
- 检索是知识复用的第一步
- 标题模糊 = 卡片永沉海底
- 信息密度低 = 维护成本高

---

### 🎯 法则二：语义纯粹原则（保持卡片纯粹）

**定义**：每张卡片只回答一个问题，只承载一种语义类型。

**检验标准**：
- **单一语义**：一张卡只属于一种类型（判断/方法/案例/信息）
- **禁止混合**：案例中的方法必须拆分，数据中的解读必须分离
- **无连接词**：不使用"以及"、"另外"、"同时"等连接词
- **无指代**：不使用"上文提到"、"前述"等上下文指代

**执行规则**：
```
遇到混合内容 → 立即拆分
1. 识别主类型（判断/方法/案例）
2. 提取嵌套的其他类型
3. 用 related 字段建立链接
```

**反模式示例**：
- ❌ 案例卡中内嵌方法步骤 → 拆成两张卡
- ❌ "传播很重要，而且要降低门槛" → 拆成两张独立卡
- ✅ 案例归案例，方法单独抽

**Why（为什么）**：
- 混合语义 = 无法独立复用
- 语义纯粹是原子化的基础
- 链接优于混合

---

### 🔮 法则三：未来复用原则（剥离特定语境）

**定义**：每张卡片必须能脱离原文、跨时间、跨场景复用。

**检验标准**：
- **去语境化**：删除"本文作者"、"这篇文章"等限定词
- **抽象化表达**：将具体场景抽象为通用原则
- **条件前置**：把特定场景转化为"适用条件"字段
- **来源元数据化**：来源信息放入 frontmatter 的 sources 字段

**抽象化技巧**：
```
具体 → 通用
"抖音的算法推荐机制" → "算法推荐产品的冷启动策略"
"Nike的营销案例" → "品牌危机下的营销重启方法论"
"2026年短视频仍有红利" → "垂直小号流量红利判断"
   + 适用条件：C端内容产品、2026年时间窗口
```

**反模式示例**：
- ❌ "根据本文作者的观点，短视频在2026年仍有机会"
- ✅ "垂直小号在2026年仍有流量红利"
   - 适用条件：C端内容产品、冷启动阶段
   - 来源：[具体文章链接放metadata]

**Why（为什么）**：
- 特定语境是复用的障碍
- 知识资产必须可迁移
- Metadata 承载来源，内容承载原则

---

### 三大法则的综合检验流程

```
每创建一张卡，通过三层检验：

Layer 1: 精确检索检验
├─ 标题是完整陈述句吗？
├─ 包含几个核心概念？(>1则拆分)
└─ 长度是否 ≤ 300字？

Layer 2: 语义纯粹检验
├─ 只回答一个问题吗？
├─ 是否混合多种类型？
└─ 离开原文能独立理解吗？

Layer 3: 未来复用检验
├─ 是否包含特定语境词？
├─ 能否抽象为通用原则？
└─ 来源信息是否在metadata？

✓ 三层全通过 → 创建卡片
✗ 任一层失败 → 重构或拆分
```

### 三大法则的优先级

```
未来复用 > 语义纯粹 > 精确检索
```

**Why**：
- 未来复用是最终目标（知识资产的长期价值）
- 语义纯粹是手段（保证可复用性）
- 精确检索是基础（保证可发现性）

## 一、卡片分类规则（基于三大法则的细化）

### 1.1 核心判别：每张卡回答一个问题

| 卡类型 | 回答的问题 | 核心特征 | 一句话验证 |
|--------|-----------|---------|-----------|
| **判断卡** | "该怎么**判断**？" | 可辩护的**主张**，有适用条件和边界 | "有人会不同意吗？" → 是 = 判断卡 |
| **方法卡** | "具体**怎么做**？" | 可复用的**流程**，有步骤和预期产出 | "别人按这个能做出来吗？" → 是 = 方法卡 |
| **案例卡** | "**发生了**什么？" | 有**叙事结构**的实例，有因果和教训 | "能追溯到具体来源吗？" → 是 = 案例卡 |
| **信息卡** | "**是什么**？" | 可检索的**事实**，中性无主张 | "搜30秒就能找到吗？" → 是 = **别建卡** |

### 1.2 各类型必含字段

**判断卡**：主张 + 适用条件 + 不适用条件 + 信心等级

**方法卡**：前置假设 + 分步流程 + 预期产出 + 已知局限

**案例卡**：具体情境（谁/何时/何地）+ 发生了什么 + 为什么重要 + 来源可追溯

**信息卡**：极少创建，仅当"难搜索 + 高频引用 + 战略重要"三者同时满足

### 1.3 提取优先级

```
判断卡 > 方法卡 > 案例卡 >>> 信息卡（信息卡默认不建）
```

> **为什么**：判断卡是核心资产（商业护城河），方法卡是杠杆点，案例卡是证据库。信息随时可搜索，无沉淀价值。

### 1.4 各类型举例

**判断卡示例**：
- "传播力 > 功能完备性"（适用：C端社交产品；不适用：B端企业工具）
- "垂直小号在 2026 年仍有流量红利"
- "冷启动阶段应优先做社交货币而非功能完善"

**方法卡示例**：
- "三步传播产品设计心法：识别社交货币 → 降低分享门槛 → 设计传播闭环"
- "零预算爆火产品的 5 步验证法"
- "公众号精准获客的三年迭代方法"

**案例卡示例**：
- "SBTI：1个 HTML + 27 张图，零后台零数据库，服务器被挤爆"
- "Nike 'Just Do It' 营销活动如何扭转品牌颓势"

**信息卡示例（应谨慎创建）**：
- 某平台最新的算法推荐机制文档（难搜索 + 高频引用）
- 某行业的关键数据指标体系（难搜索 + 战略重要）

## 二、颗粒度标准

### 2.1 "刚刚好"的 5 条规则

**规则 1：标题测试**
标题必须是**完整的陈述句**，能概括整张卡的核心观点。如果标题只能写"关于 XX 的几点思考"，说明卡太粗了。

**规则 2：一屏原则**
卡的内容不应超过一个屏幕（约 200-500 字中文）。需要滚动的卡 = 该拆。

**规则 3：独立性原则**
卡必须**自足**——不依赖其他卡就能被理解。读不懂的孤卡 = 太碎或写得太差。

**规则 4：社交发布测试（Thread 测试）**
这张卡原封不动发到社交媒体上，别人看得懂吗？看不懂 = 重写或太碎。

**规则 5：一个主张**
并列关系（第一点、第二点、第三点）如果每点都能独立成立，就该拆成多张卡。

### 2.2 颗粒度的两端信号

**太粗了 → 该拆**：
- 标题不能一句话概括
- 包含多个不相关的观点
- 需要"第一、第二、第三"来组织
- 超过一屏需要滚动
- 内容同时回答多个问题

**太碎了 → 该合并**：
- 离开上下文就失去意义
- 短于一个完整的句子
- 拆出来的碎片之间没有链接，变成孤儿卡
- 维护成本 > 价值

### 2.3 实操原则

> "按你能记住的知识粒度来拆"（Obsidian 中文社区共识）

- 拆粗了 → 以后随时可以再拆，成本低
- 拆碎了 → 孤儿卡片维护成本高，且容易丢失
- **宁可先粗后细，不要一步到位拆太碎**

### 2.4 Andy Matuschak 的三步拆法

1. **写大卡**：抓住一簇内容的"大思想"。如果有多个大思想，就写多张卡（保持原子性）
2. **写细卡**：逐段扫描，提炼更精细的原子洞察
3. **建立连接**：搜索已有卡片，建立链接、合并、修订

## 三、分类边界处理

### 3.1 边界决策树

```
这段内容的核心目的？
│
├─ 帮我"做决定"（有主张、可辩论）
│  └─ 有适用条件吗？ → 判断卡
│  └─ 没有条件边界 → 不是有效判断，不建卡
│
├─ 告诉我"怎么做"（有步骤、可重复）
│  └─ 能跨场景复用吗？ → 方法卡
│  └─ 只针对某个具体场景 → 案例卡
│
├─ 展示"发生了什么"（有叙事、有因果）
│  └─ 有可提取的教训？ → 案例卡
│  └─ 只是个事实 → 信息卡（或丢弃）
│
└─ 只是陈述"是什么"（中性事实）
   └─ 搜索不到 + 高频引用 → 信息卡
   └─ 否则 → 不建卡
```

### 3.2 常见边界 Case 速查表

| 模糊内容 | 关键判断 | 归类 |
|---------|---------|------|
| "短视频比长视频更适合冷启动" | 可辩论的主张 | **判断卡** |
| "用 5 步法做冷启动测试" | 可复用流程 | **方法卡** |
| "SBTI 从 0 到 10 万用户的 3 个阶段" | 特定实例 + 因果 | **案例卡** |
| "TikTok 日活 15 亿" | 搜 30 秒就有 | **不建卡** |
| 案例中内嵌的方法 | 可独立复用 | **拆！案例归案例，方法单独抽** |
| 既像判断又像方法 | 主要帮你"决定"还是"执行"？ | **按主用途归类** |
| 既像案例又像信息 | 有叙事/因果/教训吗？ | **有 = 案例，没有 = 丢弃** |
| 数据 + 解读 | 数据是证据，解读是判断 | **拆！数据归案例，解读归判断** |
| 名人名言 | 有独立洞察价值吗？ | **有 = 融入判断卡；无 = 丢弃** |

### 3.3 多类型归属的 3 条处理原则

**原则 1：先拆再分**
内容同时属于两个类型时，拆成两张卡各归各类。不要用"既是判断又是方法"的复合类型。

**原则 2：按最高价值归类**
不可拆时，归入价值更高的类型：判断卡 > 方法卡 > 案例卡 > 信息卡。

**原则 3：交叉链接**
用 `related` 字段和 `wiki_concepts` 字段建立关联，不靠类型本身承载多义性。

## 四、反模式清单（基于三大法则的细化）

| 反模式 | 违反法则 | 为什么错 | 正确做法 |
|--------|---------|---------|---------|
| 原文摘抄不改写 | 法则3 | 没经过大脑，包含特定语境 | 用自己的话重写，抽象为通用原则 |
| 标题写"关于XX" | 法则1 | 无法精确检索，语义模糊 | 标题用完整陈述句概括核心观点 |
| "以及"、"另外"连接 | 法则2 | 混合多个语义，不纯粹 | 立即拆分成独立卡片 |
| 案例中内嵌方法 | 法则2 | 语义不纯粹，无法独立复用 | 拆！案例归案例，方法单独抽 |
| "本文作者认为" | 法则3 | 特定语境，阻碍未来复用 | 去除限定词，来源放metadata |
| 超过300字的长卡 | 法则1 | 信息密度低，检索困难 | 拆分或精简，保持≤300字 |
| 每段都建信息卡 | 法则2 | 信息是商品化的，无沉淀价值 | 信息卡默认不建 |
| 不建链接的孤立卡 | 法则2 | 孤儿卡是Zettelkasten最危险的产物 | 每张卡至少链接2个其他卡或Wiki页面 |
| "第一、第二、第三" | 法则2 | 并列关系说明该拆，语义不纯粹 | 拆成多张独立卡 |
| 按来源而非概念组织 | 法则3 | 来源是上下文，概念才是知识 | 按概念原子组织，来源放metadata |

### 三大法则执行检查清单

**创建每张卡前，确认：**

**精确检索检查**：
- [ ] 标题是10-25字的完整陈述句
- [ ] 标题包含核心概念词
- [ ] 正文不超过300字
- [ ] 没有"关于"、"几点"等泛化词

**语义纯粹检查**：
- [ ] 只回答一个问题
- [ ] 没有使用"以及"、"另外"、"同时"
- [ ] 没有混合多种类型
- [ ] 离开原文能独立理解

**未来复用检查**：
- [ ] 没有"本文作者"、"这篇文章"等限定词
- [ ] 具体场景已抽象为通用原则
- [ ] 特定场景转化为"适用条件"字段
- [ ] 来源信息在sources字段

## 五、与其他体系的映射

| 本系统 | 雷墨（少数派） | Nick Milo LYT | Andy Matuschak |
|--------|--------------|---------------|----------------|
| 判断卡 | 观点卡（子集） | Statements | Statement notes |
| 方法卡 | 方法卡 + 行动卡 | Things（子集） | Imperative notes |
| 案例卡 | 案例卡 | Things（子集） | 无明确类型 |
| 信息卡 | 概念卡（子集） | Things | Concept notes |

> **设计决策**：本系统有意将概念定义归入 Wiki 层（概念卡/），卡片层只沉淀价值型知识（判断/方法/案例）。这与"信息归 Wiki，判断归 Lobster"的分工一致。
