# 方法卡 (Methods)

## 定义

**笔记四层结构中用于记录"具体做法"的卡片类型，提供可复用的操作流程和标准步骤**

## 核心价值

- **解决**: 「具体做法」
- **举例**: 拆卡的标准流程 (SOP)
- **价值**: 可复用的操作流程，避免重复思考

## 在笔记体系中的位置

```
价值金字塔 (从高到低)
┌─────────────────────────┐
│   判断卡 (Judgments)     │ ⭐⭐⭐⭐⭐ 最核心
├─────────────────────────┤
│   方法卡 (Methods)       │ ⭐⭐⭐⭐ ← 这里
├─────────────────────────┤
│   案例卡 (Cases)         │ ⭐⭐⭐
├─────────────────────────┤
│   信息卡 (Information)   │ ⭐ 最低价值
└─────────────────────────┘
```

## 与判断卡的区别

| 维度 | 判断卡 | 方法卡 |
|------|--------|--------|
| **解决问题** | 怎么判断 | 具体做法 |
| **核心内容** | 决策依据 | 操作步骤 |
| **典型示例** | 什么情况下用 A 而不是 B | A/B 测试五步法 |
| **价值来源** | 高杠杆决策 | 可复用流程 |
| **应用场景** | 快速决策 | 执行操作 |

## 方法卡的核心要素

### 1. 元数据结构

```yaml
---
type: method
confidence: high/medium/low
tags: [标签1, 标签2]
status: new/growing/mature
created: 2026-04-09
updated: 2026-04-09
---
```

### 2. 内容结构

- **适用场景**: 在什么情况下使用这个方法
- **前置条件**: 使用这个方法需要满足什么条件
- **操作步骤**: 具体的执行步骤（SOP）
- **预期结果**: 按照方法执行后应该得到什么结果
- **注意事项**: 执行过程中需要注意的关键点
- **验证记录**: 这个方法被验证过多少次

## 标准操作流程 (SOP) 格式

```markdown
## 方法名称

### 适用场景
在什么情况下使用这个方法

### 前置条件
- 条件1
- 条件2

### 操作步骤

#### 步骤 1: [步骤名称]
- 具体操作
- 预期产出

#### 步骤 2: [步骤名称]
- 具体操作
- 预期产出

...

### 预期结果
描述按照方法执行后应该得到的结果

### 注意事项
- 关键点1
- 关键点2

### 验证记录
- 2026-XX-XX: 应用场景1，结果：成功
- 2026-XX-XX: 应用场景2，结果：需要调整
```

## 实践示例

**场景**: 如何创建判断卡

```markdown
---
type: method
confidence: high
tags: [笔记方法, 判断卡创建, 知识管理]
status: mature
---

# 判断卡创建方法

### 适用场景
当你需要记录一个决策判断或经验法则时

### 前置条件
- 已经经历过类似情况
- 有明确的判断依据
- 能够说出适用条件

### 操作步骤

#### 步骤 1: 识别判断机会
- 从工作和学习中识别需要做决策的场景
- 识别出可以复用的判断逻辑

#### 步骤 2: 提取判断依据
- 为什么这样判断
- 什么情况下适用
- 什么情况下不适用

#### 步骤 3: 标注信心程度
- 高: 30次以上验证
- 中: 1-2次随机验证
- 低: 未验证的想法

#### 步骤 4: 记录验证历史
- 每次应用后记录结果
- 根据反馈更新信心程度

### 预期结果
得到一张可快速检索、可复用的判断卡

### 注意事项
- 每张卡片只讲一个核心判断
- 保持原子化，不要混合多个判断
- 定期回顾低信心判断卡

### 验证记录
- 2026-03-15: 项目A应用，成功节省决策时间50%
- 2026-04-01: 团队推广，5人采纳
```

## 方法卡的类型

### 1. 流程型方法

描述一系列固定的操作步骤

**示例**: A/B 测试五步法、用户调研六步法

### 2. 检查清单型方法

列出需要检查的所有项目

**示例**: 发布前检查清单、代码审查清单

### 3. 决策树型方法

根据不同条件选择不同路径

**示例**: 技术选型决策树、功能优先级决策树

### 4. 模板型方法

提供可填写的模板结构

**示例**: 周报模板、会议纪要模板

## 信心程度校准

| 等级 | 触发条件 | 示例 |
|------|----------|------|
| **高** | 30次以上验证或广泛采用 | 经典方法论（如敏捷开发） |
| **中** | 1-2次随机验证 | 自创方法，初步验证 |
| **低** | 未验证的想法 | 理论推导，待实践 |

## 状态流转规则

- **New → Growing**: 被应用1次或被1人采纳
- **Growing → Mature**: 被应用3次以上或被多人采纳
- **Mature → Outdated**: 发现更优方法或环境变化
- **任何状态 → Discarded**: 方法被证明无效

## 与判断卡的关系

```
判断卡 ← 决定做什么
    ↓
方法卡 ← 决定怎么做
    ↓
案例卡 ← 验证是否有效
    ↓
信息卡 ← 提供支撑数据
```

**协作模式**:
- 判断卡提供决策依据
- 方法卡提供执行步骤
- 两者配合使用效果最佳

## 实践建议

1. **标准化格式**: 使用统一的元数据和结构
2. **详细程度**: 步骤要具体到可以直接执行
3. **保持更新**: 根据应用反馈持续优化
4. **版本管理**: 重要变更记录版本历史
5. **关联判断**: 每个方法卡应该关联相关的判断卡

## 分类边界：方法卡 vs 判断卡 vs 案例卡

### 方法卡的核心判别

**问自己：读者看完能否直接照着做？**

- 能 → 方法卡
- 不能（只是理解了一个道理） → 判断卡
- 不需要做（只是了解事实） → 案例卡

### 最容易混淆的场景

| 混淆场景 | 正确分类 | 判断依据 |
|---|---|---|
| "从 A 的做法中提炼出 B 方法" | **方法卡** | 提炼的是通用流程，可复用 |
| "如何复制 A 的成功" | **方法卡** | 有可执行步骤 |
| "A 的成功说明 B 逻辑是对的" | **判断卡** | 只有因果判断，没有可执行步骤 |
| "A 产品用 B 方法取得了 C 效果" | **案例卡** | 具体事件记录，非通用流程 |

### 方法卡的颗粒度

- **一张方法卡 = 一个完整流程 或 一个独立步骤**
- 如果一个流程有 5 个独立步骤，且每个步骤可单独复用 → 拆成 5 张
- 如果步骤之间强耦合，拆开没有意义 → 合为 1 张
