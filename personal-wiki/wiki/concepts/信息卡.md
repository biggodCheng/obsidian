# 信息卡 (Information)

## 定义

**笔记四层结构中价值最低的卡片类型，用于记录"是什么"的基础信息**

## 核心价值

- **解决**: 「是什么」
- **举例**: 产品参数、API 文档、定义说明
- **价值**: 杠杆率极低，可随时搜索获取

## 在笔记体系中的位置

```
价值金字塔 (从高到低)
┌─────────────────────────┐
│   判断卡 (Judgments)     │ ⭐⭐⭐⭐⭐ 最核心
├─────────────────────────┤
│   方法卡 (Methods)       │ ⭐⭐⭐⭐
├─────────────────────────┤
│   案例卡 (Cases)         │ ⭐⭐⭐
├─────────────────────────┤
│   信息卡 (Information)   │ ⭐ 最低价值 ← 这里
└─────────────────────────┘
```

**核心结论**: 信息卡的杠杆率极低，不建议大量创建，可随时搜索获取。

## 与其他卡片类型的对比

| 维度 | 判断卡 | 方法卡 | 案例卡 | 信息卡 |
|------|--------|--------|--------|--------|
| **解决问题** | 怎么判断 | 具体做法 | 真实发生的事 | 是什么 |
| **核心内容** | 决策依据 | 操作步骤 | 事实记录 | 基础数据 |
| **典型示例** | 什么情况下用 A 而不是 B | A/B 测试五步法 | 客户 A 的转化率提升过程 | 产品参数 |
| **价值等级** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐ |
| **可复用性** | 高 | 高 | 中 | 低 |
| **搜索替代性** | 难 | 中 | 中 | 易 |

## 为什么信息卡价值低？

### 1. 容易获取

- 大部分信息可以通过搜索引擎快速找到
- 官方文档、百科全书、技术手册等
- 不需要专门存储在个人知识库中

### 2. 容易过时

- 技术参数会更新
- API 会变更
- 产品会迭代
- 存储的信息很快过时

### 3. 缺乏洞察

- 信息卡只是记录"是什么"
- 没有包含"为什么"和"怎么做"
- 无法直接指导决策和行动

### 4. 占用空间

- 大量信息卡会淹没高价值的判断卡
- 降低检索效率
- 增加维护成本

## 什么时候需要创建信息卡？

### ⚠️ 谨慎创建原则

**不推荐创建**:
- ❌ 公开的 API 文档
- ❌ 可以通过搜索找到的技术参数
- ❌ 通用的定义和概念
- ❌ 容易获取的数据

**可以考虑创建**:
- ✅ 内部专有信息（无法通过搜索获取）
- ✅ 个人整理的信息架构
- ✅ 需要频繁访问的本地信息
- ✅ 与判断卡/方法卡强关联的支撑信息

### 替代方案

与其创建信息卡，不如：

1. **使用外部链接**
   - 保存官方文档链接
   - 使用书签工具管理

2. **使用引用**
   - 在判断卡中引用外部来源
   - 需要时再去查找

3. **创建判断卡**
   - 从信息中提炼判断
   - 记录"这个信息意味着什么"

## 信息卡的标准格式（如需创建）

### 1. 元数据结构

```yaml
---
type: information
confidence: high/medium/low
tags: [标签1, 标签2]
status: new/growing/mature
created: 2026-04-09
updated: 2026-04-09
---
```

### 2. 内容结构

- **信息标题**: 清晰描述信息内容
- **信息来源**: 来自哪里（官方文档、内部文档等）
- **关键数据**: 核心信息
- **更新频率**: 信息可能更新的频率
- **关联判断**: 这个信息支撑哪些判断

## 实践示例

### ❌ 不推荐的信息卡

```markdown
---
type: information
tags: [API, 文档]
---

# REST API 请求方法

### GET
用于获取资源

### POST
用于创建资源

### PUT
用于更新资源

### DELETE
用于删除资源
```

**为什么不推荐**:
- 这是公开的标准信息
- 随时可以搜索到
- 没有个人洞察
- 占用存储空间

### ✅ 可以考虑的信息卡

```markdown
---
type: information
tags: [内部系统, API]
source: 内部技术文档
last_updated: 2026-04-09
---

# 内部订单系统 API 限制

### 请求频率限制
- 每秒最多 100 次请求
- 超过限制返回 429 错误

### 数据格式要求
- 必须使用 JSON 格式
- 字段名称使用驼峰命名

### 特殊注意事项
- 订单创建接口有 5 秒的防重放窗口
- 退款接口需要额外权限

### 关联判断
- [[API 限流策略]]
- [[系统稳定性判断]]
```

**为什么可以考虑**:
- 内部专有信息，外部无法获取
- 需要频繁访问
- 关联到具体的判断卡

## 信息卡 vs 判断卡

### 信息卡示例

```markdown
# 产品 A 的价格是 $99/月
```

**价值**: 只是记录了一个事实，无法直接指导决策

### 判断卡示例

```markdown
# $99/月的定价策略适合早期采用者市场

## 判断依据
- 早期采用者对价格不敏感
- 高价格可以筛选出高质量用户
- 为后续降价留出空间

## 适用条件
- 产品有独特价值
- 竞争对手少
- 目标用户是 B2B
```

**价值**: 提供了决策依据，可以直接指导行动

## 核心原则

> **"从信息到洞察，是知识管理的核心跃迁"**

- **信息**: 产品价格是 $99/月
- **洞察**: $99/月的定价适合早期采用者市场

**优先创建判断卡，而非信息卡**

## 实践建议

1. **谨慎创建**: 除非信息无法通过搜索获取，否则不创建
2. **及时删除**: 定期清理过时或不再需要的信息卡
3. **关联判断**: 如果必须创建，要关联到相关的判断卡
4. **提炼判断**: 从信息中提炼判断，创建判断卡
5. **使用外部工具**: 使用书签、浏览器收藏夹等管理外部信息

## 信息卡的替代方案

### 方案 1: 外部链接

在判断卡中保存外部链接：

```markdown
## 参考资源
- 官方文档: https://example.com/docs
- API 参考: https://example.com/api
```

### 方案 2: 书签工具

使用专门的书签管理工具：
- 浏览器书签
- Raindrop.io
- Pocket
- Notion Web Clipper

### 方案 3: 搜索技能

提升搜索技能，快速找到需要的信息：
- 学习高级搜索语法
- 建立常用信息源清单
- 使用专业搜索工具
