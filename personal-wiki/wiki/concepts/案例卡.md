# 案例卡 (Cases)

## 定义

**笔记四层结构中用于记录"真实发生的事"的卡片类型，提供验证过的事实依据**

## 核心价值

- **解决**: 「真实发生的事」
- **举例**: 客户 A 的转化率提升过程
- **价值**: 验证过的事实依据，支撑判断和方法

## 在笔记体系中的位置

```
价值金字塔 (从高到低)
┌─────────────────────────┐
│   判断卡 (Judgments)     │ ⭐⭐⭐⭐⭐ 最核心
├─────────────────────────┤
│   方法卡 (Methods)       │ ⭐⭐⭐⭐
├─────────────────────────┤
│   案例卡 (Cases)         │ ⭐⭐⭐ ← 这里
├─────────────────────────┤
│   信息卡 (Information)   │ ⭐ 最低价值
└─────────────────────────┘
```

## 与其他卡片类型的区别

| 维度 | 判断卡 | 方法卡 | 案例卡 |
|------|--------|--------|--------|
| **解决问题** | 怎么判断 | 具体做法 | 真实发生的事 |
| **核心内容** | 决策依据 | 操作步骤 | 事实记录 |
| **典型示例** | 什么情况下用 A 而不是 B | A/B 测试五步法 | 客户 A 的转化率提升过程 |
| **价值来源** | 高杠杆决策 | 可复用流程 | 验证过的事实 |
| **应用场景** | 快速决策 | 执行操作 | 验证观点 |

## 案例卡的核心要素

### 1. 元数据结构

```yaml
---
type: case
confidence: high/medium/low
tags: [标签1, 标签2]
status: new/growing/mature
created: 2026-04-09
updated: 2026-04-09
---
```

### 2. 内容结构

- **案例背景**: 时间、地点、人物、环境
- **问题/挑战**: 面临什么问题或挑战
- **行动方案**: 采取了什么行动
- **结果**: 最终结果如何（最好有数据）
- **关键洞察**: 从这个案例中学到什么
- **验证记录**: 这个案例被引用过多少次

## 案例卡的类型

### 1. 成功案例

记录成功的经验和做法

**示例**: 
- 通过优化首页设计，转化率提升 35%
- 采用新架构后，系统性能提升 10 倍

### 2. 失败案例

记录失败的教训和坑

**示例**:
- 错误的技术选型导致项目延期
- 忽略用户体验导致产品失败

### 3. 实验案例

记录 A/B 测试等实验结果

**示例**:
- 测试两种定价策略，A 方案效果更好
- 测试三种营销渠道，渠道 C ROI 最高

### 4. 观察案例

记录观察到的现象和模式

**示例**:
- 观察到用户在特定场景下的异常行为
- 发现竞品的某个策略效果显著

## 实践示例

**场景**: 定价策略优化案例

```markdown
---
type: case
confidence: high
tags: [定价策略, A/B测试, 增长]
status: mature
---

# SaaS 产品定价策略优化案例

### 案例背景
- **时间**: 2026年1-3月
- **公司**: B2B SaaS 公司
- **产品**: 企业协作工具
- **环境**: 用户增长放缓，需要优化收入

### 问题/挑战
- 月费用户增长缓慢
- 竞争对手降价压力
- 需要提升收入

### 行动方案
1. 设计三种定价方案进行 A/B 测试
   - 方案A: 保持月费，增加年费优惠
   - 方案B: 降低月费，取消年费
   - 方案C: 保持月费，增加季度选项
2. 运行测试 4 周
3. 收集数据并分析

### 结果
- **量化结果**: 
  - 方案A: 年费转化率提升 40%，收入增长 25%
  - 方案B: 用户数增长 30%，但收入下降 10%
  - 方案C: 无明显变化
- **质化结果**: 用户更倾向于年费承诺

### 关键洞察
1. B2B 用户更看重长期承诺而非短期价格
2. 年费优惠是有效的增长策略
3. A/B 测试必须结合收入和用户数两个指标

### 关联判断
- [[定价策略判断]]
- [[B2B vs B2C 定价差异]]

### 验证记录
- 2026-04-01: 被引用于新产品定价决策
- 2026-04-05: 被分享给其他团队
```

## 信心程度校准

| 等级 | 触发条件 | 示例 |
|------|----------|------|
| **高** | 有明确数据支撑，可重复验证 | 有完整数据的实验案例 |
| **中** | 有一定依据，但数据不完整 | 观察到现象，但缺乏详细记录 |
| **低** | 道听途说，未经验证 | 他人提及，无第一手资料 |

## 案例卡的价值

### 1. 验证判断和方法

- 判断卡说"X 方法有效"
- 案例卡提供"X 方法在项目 A 中确实有效的具体证据"

### 2. 提供具体情境

- 理论需要结合实际情境
- 案例卡提供丰富的情境信息

### 3. 积累经验教训

- 成功案例可以复制
- 失败案例可以避免重蹈覆辙

### 4. 支撑决策

- 用事实说话，而非主观臆断
- 提供数据支撑，增强说服力

## 与其他卡片的协作

```
信息卡 ← 提供基础数据
    ↓
案例卡 ← 记录事实经过
    ↓
判断卡 ← 提炼决策经验
    ↓
方法卡 ← 总结通用方法
```

**协作模式**:
- 案例卡提供原始素材
- 判断卡从案例中提炼经验
- 方法卡从案例中总结流程
- 信息卡提供案例的背景数据

## 实践建议

1. **记录及时**: 案例发生后尽快记录，避免遗忘细节
2. **数据优先**: 尽可能记录量化数据，而非主观感受
3. **情境完整**: 记录足够的背景信息，便于后续理解
4. **关联判断**: 明确这个案例验证或反驳了哪些判断
5. **定期回顾**: 定期回顾案例，提炼新的判断和方法

## 案例研究的深度层次

| 层次 | 描述 | 价值 |
|------|------|------|
| **浅层** | 简单记录发生了什么 | ⭐ 基础参考 |
| **中层** | 分析为什么发生 | ⭐⭐ 理解原因 |
| **深层** | 提炼可复用的模式 | ⭐⭐⭐ 形成判断 |
| **系统** | 建立案例之间的连接 | ⭐⭐⭐⭐ 知识网络 |

## 分类边界：案例卡 vs 方法卡 vs 判断卡

### 案例卡的核心判别

**问自己：这是在记录"发生了什么"，还是在讲"怎么做到"？**

- 记录发生了什么（具体对象+具体结果） → 案例卡
- 讲怎么做到（通用流程+可复用） → 方法卡
- 讲为什么（因果推理+有观点） → 判断卡

### 最容易混淆的场景

| 混淆场景 | 正确分类 | 判断依据 |
|---|---|---|
| "A 产品用 B 方法取得了 C 效果" | **案例卡** | 有具体对象+具体结果，可验证 |
| "分析 A 的设计模式" | **方法卡** | 提炼的是通用设计模式，非具体事件 |
| "A 的成功说明 B 逻辑是对的" | **判断卡** | 提炼的是因果判断 |

### 案例卡的颗粒度

- **一张案例卡 = 一个完整事件**（背景→过程→结果）
- 必须有具体对象（谁/哪个产品）
- 必须有具体结果（数据/效果）
- 如果没有具体对象和结果，不应该标为案例卡
